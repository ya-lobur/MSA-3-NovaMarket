# Task 2: Агрегированные данные на фронтенде - История заказов

Это решение второго задания для реализации функции просмотра истории заказов с агрегированными данными из нескольких микросервисов.

## Структура решения

### 1. Обновленная архитектурная схема приложения (C2 диаграмма)

**Файл:** [diagrams/puml/c2-architecture-updated.puml](diagrams/puml/c2-architecture-updated.puml)

![Updated C2 Architecture](diagrams/img/c2-architecture-updated.png)

#### Ключевые изменения в архитектуре:

Добавлен новый микросервис **Order History Service** для реализации паттерна CQRS:

- **Order History Service** - микросервис для чтения истории заказов с агрегированными данными
  - Имеет собственную базу данных (Order History DB) с денормализованными данными
  - Подписывается на все релевантные события через Event Bus
  - Обновляет материализованные представления при получении событий
  - Предоставляет быстрый Read API для клиентов

#### Принцип работы:

1. **Write Model (Command)**: Order Service и другие сервисы создают и изменяют заказы, публикуя события
2. **Event Bus**: Передает события всем подписчикам
3. **Read Model (Query)**: Order History Service обрабатывает события и обновляет денормализованные данные
4. **Client**: Запрашивает историю напрямую из Order History Service без обращения к другим сервисам

### 2. ADR (Architecture Decision Record)

**Файл:** [adr-order-history.md](adr-order-history.md)

ADR содержит детальное обоснование архитектурного решения:

#### Выбранное решение: **CQRS с материализованными представлениями**

**Обоснование:**
- ✅ Удовлетворяет требованиям к производительности (< 300ms в 98%, < 1s в 99.99%)
- ✅ Высокая доступность (не зависит от других сервисов при чтении)
- ✅ Масштабируемость до 40,000+ заказов в день
- ✅ Легкая расширяемость для добавления новых данных
- ✅ Согласуется с EDA и микросервисной архитектурой

#### Рассмотренные альтернативы:

1. **API Composition**
   - ❌ Не удовлетворяет требованиям к производительности
   - ❌ Зависимость от доступности всех сервисов
   - ✅ Простота реализации
   - ✅ Strong consistency

2. **Event Sourcing**
   - ❌ Избыточная сложность для данного use case
   - ❌ Проблемы с производительностью чтения
   - ✅ Полная история изменений
   - ✅ Audit trail

3. **Database View в Order Service**
   - ❌ Нарушение Bulkhead Pattern
   - ❌ Coupling между сервисами
   - ✅ Простота реализации

#### Trade-offs выбранного решения:

**Недостатки:**
- Eventual consistency (задержка 1-2 секунды)
- Дублирование данных
- Сложность синхронизации через события

**Митигация:**
- Transactional Outbox Pattern для гарантии доставки
- Идемпотентные обработчики событий
- Kafka с гарантией порядка
- Мониторинг задержки обработки событий

## Ключевые архитектурные паттерны

Решение использует следующие паттерны:

### 1. CQRS (Command Query Responsibility Segregation)
- Разделение моделей для записи (Command) и чтения (Query)
- Order Service - команды создания/изменения заказов
- Order History Service - запросы истории заказов

### 2. Materialized Views
- Денормализованные представления данных в Order History DB
- Агрегация данных из Order, Catalog, Payment, Delivery, Inventory
- Быстрое чтение без JOIN-ов и вызовов других сервисов

### 3. Event-Driven Architecture (EDA)
- Асинхронная синхронизация данных через события
- Order History Service подписан на: `OrderCreated`, `ItemsReserved`, `PaymentSucceeded`, `DeliveryCreated`, `DeliveryCompleted`, `OrderCancelled`, `ProductUpdated`, etc.

### 4. Transactional Outbox Pattern
- Гарантия доставки событий
- Атомарность бизнес-операции и публикации события

### 5. Database per Service
- Order History Service имеет собственную БД
- Изоляция данных и независимое масштабирование

## Соответствие требованиям

### Функциональные требования:
✅ Просмотр списка всех заказов покупателя
✅ Детальная информация о заказе с агрегированными данными
✅ Данные о товарах, ценах, оплате, доставке, статусах
✅ Готовность к добавлению функций: скачивание чека, повторный заказ, отзывы

### Нефункциональные требования:
✅ Производительность: < 300ms (98%), < 1s (99.99%) - достигается через денормализацию
✅ Масштабируемость: готовность к 40,000+ заказов в день
✅ Доступность: чтение не зависит от других сервисов
✅ Согласованность: eventual consistency с задержкой < 2 секунды
✅ Расширяемость: легко добавить новые поля через подписку на события

## Технологический стек

- **База данных**: PostgreSQL с JSONB для гибкого хранения
- **Event Bus**: Kafka для надежной доставки событий
- **Кэширование**: Redis для часто запрашиваемых данных
- **API**: REST API через API Gateway
- **Оркестрация**: Kubernetes с HPA для автоскейлинга
- **Мониторинг**: Prometheus + Grafana для метрик обработки событий

## Мониторинг и операционная поддержка

### Ключевые метрики:
- Event processing lag (задержка обработки событий)
- Query response time (время ответа на запросы)
- Event delivery success rate (успешность доставки событий)
- Database query performance (производительность БД)

### Алертинг:
- Задержка обработки событий > 5 секунд
- Failed event processing
- Query response time > 500ms (p95)

## Дальнейшее развитие

Архитектура готова к добавлению новых функций:

1. **Скачивание чека** - Order History Service может генерировать PDF на основе имеющихся данных
2. **Повторный заказ** - использование данных из истории для создания нового заказа
3. **Отзывы** - новый Review Service может публиковать события, обогащающие историю
4. **Аналитика** - новые сервисы могут подписаться на те же события для своих целей
5. **Программа лояльности** - расчет бонусов на основе истории заказов
