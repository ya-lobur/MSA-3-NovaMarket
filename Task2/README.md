# Задание 2. Получите агрегированные данные на фронтенде

Бизнес хочет развивать сервис, так что теперь перед вами стоит задача реализовать сценарий, в рамках которого у
пользователя появится возможность просматривать всю историю своих заказов.

---

## Подробное описание сценария «Просмотр истории покупок»

После завершения доставки покупатель может вернуться в личный кабинет, чтобы посмотреть историю покупок.

- В списке отображаются все заказы: сколько они стоили, когда были сделаны и каким способом доставлены.
- Для каждого заказа доступна подробная информация: список товаров, их количество и цена, итоговая сумма, способ
  доставки
  и оплаты, финальный статус заказа (доставлен, отменён).
- Покупатель может скачать чек, оставить отзыв о товаре или заказать его снова.

Также стоит учитывать требования к отклику всех интерфейсов приложения.

---

## Что нужно сделать

1. Составьте контейнерную (С2) диаграмму в нотации С4 и отразите в ней обновленную архитектуру приложения, в которой вы
   реализуете описанные требования.
2. Составьте ADR. Для реализации вам нужно будет применить один из паттернов или их комбинацию: API Composition, CQRS,
   Event Sourcing.

В рамках ADR опишите:

- какой вариант решения вы выбрали,
- какие альтернативы рассматривали,
- в чём их плюсы и минусы,
- последствия для вашего решения.

Добавьте всё это в соответствующие блоки ADR.

Чтобы сдать решение в формате ADR, используйте [шаблон](task_src/adr_template.md).

Выполните это задание с помощью PlantUML или любого другого удобного для вас инструмента.

Схемы и диаграммы могут быть в виде как PlantUml, так и простых картинок в формате PNG или JPG.

В результате у вас должно получиться:

- Контейнерная диаграмма (C2) с обновлённой архитектурой приложения;
- ADR.
