# Задание 4. Повышение надёжности приложения

## Часть 1. Rate Limiter

### Проблема

В исходной конфигурации все клиенты (веб и мобильные приложения) использовали одну зону с лимитом 100 r/s, что не
позволяло контролировать нагрузку от разных каналов.

### Решение

Создано две отдельные зоны:

- **web_zone**: 50 запросов/сек с одного IP для веб-приложения
- **mobile_zone**: 30 запросов/сек с одного IP для мобильного приложения

Разделение производится через разные эндпоинты:

- `/api/web/` — для веб-приложения
- `/api/mobile/` — для мобильного приложения

Параметр `burst` установлен пропорционально лимиту (в 2 раза больше базового rate) для обработки кратковременных
всплесков нагрузки.

### Альтернативный подход

Если клиенты не могут разделяться по путям, можно использовать заголовок (например, `X-Client-Type`):

```nginx
map $http_x_client_type $limit_key {
    "web"    $binary_remote_addr;
    "mobile" $binary_remote_addr;
    default  "";
}

limit_req_zone $limit_key zone=web_zone:10m rate=50r/s;
limit_req_zone $limit_key zone=mobile_zone:10m rate=30r/s;
```

## Часть 2. Circuit Breaker

### Проблема

Исходная конфигурация имела слишком большие таймауты (30 секунд), что приводило к долгому ожиданию при проблемах с
внешним сервисом.

### Решение

Реализован паттерн Circuit Breaker:

1. **Таймауты сокращены до 3 секунд:**
    - `proxy_connect_timeout 3s` — время на установку соединения
    - `proxy_read_timeout 3s` — время ожидания ответа
    - `proxy_send_timeout 3s` — время на отправку запроса

2. **Автоматическое отключение проблемного upstream:**
    - `max_fails=5` — после 5 неудачных запросов
    - `fail_timeout=30s` — сервер считается недоступным на 30 секунд

3. **Fallback-ответ:**
    - При ошибках 502, 503, 504 возвращается контролируемый JSON-ответ с кодом 503
    - Пользователь получает понятное сообщение вместо timeout

### Как работает Circuit Breaker

- **Closed** (норма): запросы идут к сервису
- **Open** (проблемы): после 5 ошибок подряд NGINX перестаёт отправлять запросы на 30 секунд и сразу возвращает fallback
- **Half-Open** (восстановление): через 30 секунд NGINX пробует снова отправить запрос. Если успешно — circuit
  закрывается, если нет — снова open на 30 секунд
